Informe de Mejoras — Spolt (Segunda Entrega)

Proyecto: Spolt — Red Social Deportiva

Autor: David Fernández Bueno

1. Introducción

Este documento recoge las mejoras implementadas en el proyecto Spolt desde la primera entrega hasta la segunda entrega. Se detallan los nuevos módulos incorporados tanto en el backend (NestJS) como en el frontend (Angular), las ampliaciones del esquema de base de datos, las mejoras de seguridad, la integración con servicios externos y las mejoras en la experiencia de usuario.

3. Nuevos módulos del backend

3.1. Módulo de Email (email)

Archivos: email.module.ts, email.service.ts, templates/

Se ha implementado un servicio de correo electrónico integrado con @nestjs-modules/mailer que ofrece las siguientes funcionalidades:

  • Email de bienvenida (welcome.hbs): se envía automáticamente al registrarse un nuevo usuario.
  • Email de restablecimiento de contraseña (password-reset.hbs): genera un enlace seguro con token criptográfico (crypto.randomBytes) para que el usuario pueda restablecer su contraseña.
  • Email de confirmación de cambio de correo (email-change.hbs): envía un correo de verificación al nuevo email del usuario antes de realizar el cambio.

3.2. Módulo de Meteorología (weather)

Archivos: weather.module.ts, weather.service.ts, weather.controller.ts

Se ha integrado la API de AEMET (Agencia Estatal de Meteorología) para ofrecer previsiones meteorológicas a 7 días basadas en la ubicación del usuario:

  • Precarga de municipios: al arrancar el servidor se descarga el catálogo completo de municipios españoles desde AEMET y se almacena en memoria para búsquedas rápidas.
  • Geolocalización por proximidad: se utiliza la fórmula de Haversine para calcular la distancia real entre las coordenadas del usuario y los municipios, devolviendo el más cercano.
  • Transformación de datos: los datos brutos de AEMET se transforman en un formato limpio y consumible por el frontend.
  • Traducción de iconos: los códigos oficiales de estado del cielo de AEMET se mapean a emojis representativos.
  • Fallback robusto: si la API falla o no está configurada, se utiliza Madrid como ubicación por defecto.

3.3. Módulo de Almacenamiento (storage)

Archivos: storage.module.ts, storage.service.ts, storage.controller.ts

Se ha implementado un sistema completo de almacenamiento de archivos mediante MinIO (compatible con S3):

  • Subida de archivos (POST /storage/upload): endpoint protegido con JWT que permite subir imágenes con límite de 15 MB. Validación mediante ParseFilePipe.
  • Servido de archivos (GET /storage/:objectName): endpoint público que devuelve el archivo directamente como stream.
  • Creación automática de bucket: al iniciar el módulo se asegura la existencia del bucket configurado.

3.4. Módulo de Niveles Deportivos (sport-level)

Archivos: sport-level.module.ts, sport-level.service.ts, sport-level.controller.ts

Se ha creado un sistema de gamificación con niveles deportivos por deporte:

  • Sistema de experiencia (XP): cada tipo de evento otorga diferente cantidad de XP (Partido: 50, Torneo: 150, Playday: 150).
  • Niveles progresivos: el XP necesario para subir es dinámico (51 + nivel_actual). Máximo nivel 100.
  • Upsert inteligente: se crea o actualiza automáticamente el registro al recibir XP.
  • Tracking completo: nivel, XP actual, XP total y partidos jugados.

4. Mejoras en módulos existentes

4.1. Módulo de Autenticación (auth)

  • Recuperación de contraseña: POST /auth/forgot-password y POST /auth/reset-password para flujo seguro con tokens temporales.
  • Cambio de correo electrónico: POST /auth/request-email-change y POST /auth/confirm-email-change para verificar propiedad del nuevo email.
  • Mejoras de seguridad: tracking de ultimo_acceso y validación estricta con DTOs.

4.2. Módulo de Eventos (events)

  • Finalización manual (PATCH /events/:id/finalizar): el creador puede finalizar el evento y repartir XP si está lleno y la fecha ya pasó.
  • Finalización automática (@Cron): tarea diaria a las 23:59 que procesa y finaliza eventos pasados automáticamente, repartiendo XP.
  • Mejoras de seguridad: protección total de endpoints de modificación y eliminación con JWT.

5. Cambios en el esquema de base de datos

5.1. Nueva tabla niveles_deportivos

model NivelDeportivo {
  id_nivel_deportivo  Int      @id @default(autoincrement())
  id_usuario          Int
  id_deporte          Int
  nivel               Int      @default(1)
  experiencia_actual  Int      @default(0)
  experiencia_total   Int      @default(0)
  partidos_jugados    Int      @default(0)
  fecha_actualizacion DateTime @updatedAt

  @@unique([id_usuario, id_deporte])
}

5.2. Nuevos campos en usuarios

  • reset_token: VARCHAR(255) temporal para restablecimientos.
  • reset_token_expires: DateTime.
  • new_email: VARCHAR(100) almacenamiento temporal del email propuesto.

6. Mejoras en el frontend

  • Página de Inicio: Nuevo Widget de Meteorología por geolocalización y Calendario Deportivo interactivo.
  • Página de Perfil: Sistema de pestañas dividiendo Perfil personal y Niveles Deportivos. Flujo visual de subida de Avatar.
  • Página de Eventos: Dividida en "Mis Eventos" (con ordenación y finalización) y "Unirse a Eventos" (exploración de amigos).
  • Nuevas páginas Auth: Vistas para reset de contraseña y confirmación de email.
  • Diseño Neo-Brutalist: Perfeccionamiento del sistema de diseño visual con paletas consistentes, micro-animaciones y sombras duras.

7. Mejoras de arquitectura y calidad de código

  • Programación de tareas: @nestjs/schedule para CRON jobs de finalización de eventos.
  • Transacciones Prisma: Uso de $transaction para unirse/salir de eventos, previniendo condiciones de carrera.
  • Validación con DTOs: Ampliación de validadores (ej: @Min, @Max) en entradas de API.
  • Estado reactivo: Uso de BehaviorSubject en Angular para actualizar la interfaz (ej: Avatar) instantáneamente sin recargas.
